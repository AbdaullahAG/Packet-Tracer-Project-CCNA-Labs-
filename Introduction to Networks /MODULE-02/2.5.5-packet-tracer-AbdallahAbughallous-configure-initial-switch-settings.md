# Lab: Basic Switch Configuration

## Part 1: Verify the Default Switch Configuration
### Step 2: Examine the current switch configuration (Using the `show running-config` command)

* **How many Fast Ethernet interfaces does the switch have?**
    * 24
* **How many Gigabit Ethernet interfaces does the switch have?**
    * 2
* **What is the range of values shown for the vty lines?**
    * 0 - 15
* **Which command will display the current contents of non-volatile random-access memory (NVRAM)?**
    * `show startup-config`
* **Why does the switch respond with “startup-config is not present?”**
    * Because the configuration file has not been saved to NVRAM yet. It currently exists only in RAM (running-config), which is the factory default state for a switch.

## Part 2: Create a Basic Switch Configuration
### Step 2: Secure access to the console line

* **Why is the `login` command required?**
    * The `login` command is required to enable password verification. Without this command, the switch will not prompt for a password even if one has been set.

### Step 7: Verify that the enable secret password is added

* **What is displayed for the enable secret password?**
    * Something similar to: `$1$mERr$ILwq/b7kc.7X/ejA4Aosn0` (an encrypted format).
* **Why is the enable secret password displayed differently from what we configured?**
    * Because `enable secret` always uses strong encryption (MD5 hashing) automatically, whereas the `enable password` remains in plain text unless manually encrypted.

### Step 8: Encrypt the enable and console passwords

* **If you configure any more passwords on the switch, will they be displayed in the configuration file as plain text or in encrypted form? Explain.**
    * They will be displayed in **encrypted form**.
    * The `service password-encryption` command encrypts all current and future passwords (including console, enable password, vty, etc.) using a simple encryption algorithm (Type 7). No new passwords will appear as plain text after this command is executed.

## Part 3: Configure a MOTD Banner
### Step 1: Configure a message of the day (MOTD) banner

* **When will this banner be displayed?**
    * The banner appears whenever someone accesses the switch via the console port or VTY (Telnet/SSH) before the "User Access Verification" screen appears.
* **Why should every switch have a MOTD banner?**
    * To legally warn unauthorized users that access is prohibited and to provide important messages to network technicians or administrators. It also helps document ownership and warns against unauthorized access.

## Part 4: Save Configuration Files to NVRAM
### Step 1: Save the configuration

* **What is the shortest, abbreviated version of the `copy running-config startup-config` command?**
    * `copy run start` (or even `cop r s`)
* **Which command will display the contents of NVRAM?**
    * `show startup-config`
* **Are all the changes that were entered recorded in the file?**
    * Yes, all changes (hostname, passwords, banner, service password-encryption, etc.) should be recorded in the `startup-config` file exactly as they are in the `running-config`.

## Part 5: Configure S2
Configure Switch **S2** following the same steps as S1, using the following parameters:

* **Device name:** ```bash
    hostname S2
    ```
* **Console password:** ```bash
    line console 0
    password letmein
    login
    ```
* **Privileged mode passwords:** ```bash
    enable password c1$c0
    enable secret itsasecret
    ```
* **MOTD Banner:** ```bash
    banner motd "This is a secure system. Authorized Access Only!"
    ```
* **Encrypt all plain text passwords:** ```bash
    service password-encryption
    ```
* **Verify the configuration:** ```bash
    show running-config
    ```
* **Save the configuration:** ```bash
    copy running-config startup-config
    ```

> **Tip:** After finishing S2, ensure you can `exit` and log back in using the assigned passwords (Console: `letmein`, Enable Secret: `itsasecret`).
